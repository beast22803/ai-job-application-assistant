FIELD-BY-FIELD GENERATION RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

header
  → Format: "Full Name | Email | Phone | Location | LinkedIn"
  → Only include fields present in the original resume. Omit fields that are not present.
  → Use " | " as separator.
  → Example: "John Doe | john.doe@email.com | +1 555 123 4567 | San Francisco, CA | linkedin.com/in/johndoe"
  → CRITICAL: Under no circumstances should you copy the name "John Doe", email "john.doe@email.com", or phone "+1 555 123 4567" from this example. You MUST extract the name, email, phone, and contact details from the "Original Resume" data below.

summary
  → 2–4 sentences. Maximum 60 words.
  → Write in third person or first person depending on the original resume's style.
  → Must mention: years of experience, primary technology stack, and the type of work done.
  → Integrate ATS keywords NATURALLY. Do not artificially stuff keywords where they don't belong.
  → Base ONLY on the original resume's summary and experience sections.
  → Apply memory preferences (e.g. if past_preferences suggest concise summaries, keep it 2 sentences).
  → Do NOT mention skills the candidate does not have.
  → Do NOT use "passionate", "dynamic", "results-driven", or other filler adjectives.

skills (array of strings)
  → List individual skills as short strings (e.g. "TypeScript", "Angular", "REST APIs").
  → Ordering rule: matched_skills from the job first, then remaining resume skills.
  → Apply any accepted suggestion that instructs reordering.
  → Deduplicate: if the same skill appears twice under different names, keep the canonical form.
  → Include all skills from the original resume's skills list AND any skills requested to be added by the approved suggestions (e.g., if a suggestion asks to add a skill, you MUST add it to the skills array).
  → Maximum 20 skills.

experience (array of strings)
  → Each entry is a single formatted string for one role.
  → CRITICAL: You must only include actual work experience entries that exist in the original resume's experience section. Do NOT include education entries, degrees, universities, or projects under the experience section. Keep the experience section strictly 1-to-1 with the candidate's actual job history in the original resume.
  → Format each string as:
    "JOB_TITLE · COMPANY · START_DATE–END_DATE\n- BULLET_1\n- BULLET_2\n- BULLET_3"
  → CRITICAL: Do NOT output literal square brackets `[` or `]` anywhere in the values. They are placeholders. For example, output "Full Stack Engineer · Fresh Prints · 2025–2026" NOT "[Full Stack Engineer] at [Fresh Prints]".
  → Write 2–4 bullet points per role.
  → BULLET POINT RULES (strict):
    - Start with a strong action verb from this list ONLY:
      Built, Developed, Designed, Implemented, Engineered, Delivered, Optimised, Refactored, Integrated, Automated, Reduced, Improved, Led, Mentored, Deployed, Migrated, Architected, Tested, Debugged, Configured
    - Ensure a diverse use of action verbs. Do not start every bullet with "Built" or "Developed".
    - Follow this pattern: Action Verb + specific technology/tool + what was achieved or delivered
    - Include a measurable outcome ONLY if one is explicitly stated in the original resume. Do not invent them.
    - Integrate ATS keywords naturally into the descriptions without sounding robotic.
    - Do NOT use passive voice ("was responsible for", "helped with").
    - REWRITE existing bullet points to be more impactful — do NOT copy them verbatim unless they already follow the pattern above.
  → Apply accepted suggestions that instruct rewording of specific roles.
  → If a role has no description in the original resume, write 1 bullet summarising the title and company only.
  → If experience array is empty in original resume: return [].

projects (array of strings)
  → Each entry is a single formatted string for one project.
  → Format: "PROJECT_NAME — TECH_STACK_SEPARATED_BY_COMMAS\nDESCRIPTION_TEXT"
  → CRITICAL: Do NOT output literal square brackets `[` or `]` anywhere in the values.
  → Description must use original resume language — do not fabricate new achievements.
  → Apply any accepted suggestion that instructs adding project emphasis.
  → CRITICAL: Do NOT use placeholder values like "PROJECT_NAME", "TECH_STACK" or "DESCRIPTION_TEXT" in the output. Map the actual projects from the original resume data. If there are no projects in the original resume, return an empty array [].

education (array of strings)
  → Each entry is a single formatted string.
  → Format: "DEGREE · INSTITUTION · YEAR"
  → CRITICAL: Do NOT output literal square brackets `[` or `]` anywhere in the values.
  → Example: "M.S. Computer Science · Stanford University · 2020–2022 | GPA: 3.9/4.0"
  → CRITICAL: Under no circumstances should you copy "Stanford University" or "Osmania University" from this example unless they are present in the "Original Resume". Map the actual education entries (all of them) from the "Original Resume" data below. Do not omit any degree or institution from the original resume list (e.g. if the original resume lists both a Master's and a Bachelor's, include both in the array).
  → Include GPA only if present in original resume.
  → If education array is empty in original resume: return [].

Rules:

  → "Reorder skills" suggestion: apply the reordering.
  → "Rewrite [ROLE] bullets" suggestion: rewrite that role's bullets using the action verb pattern.
  → "Add [KEYWORD] to experience" suggestion: add the keyword/technology to the relevant experience description or bullets.

STRICT ANTI-HALLUCINATION RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2. Do NOT invent job titles, companies, dates, projects, degrees, or institutions.
3. Do NOT invent metrics (percentages, dollar amounts, user counts, team sizes) unless they appear verbatim in the original resume.
4. Do NOT add a skill to the skills array that is not in the original resume's skills list OR present in the approved suggestions list.
5. Do NOT create a project entry that is not in the original resume's projects list.
6. Do NOT create an experience entry that is not in the original resume's experience list.

8. Do NOT use null for any field — use "" for empty strings and [] for empty arrays.
9. Do NOT add commentary, notes, or warnings inside the JSON.
10. NEVER change the company names of the candidate's past work experiences to the target company name. All past companies and employers must remain exactly as they appear in the original resume. Do NOT change them.
11. Do NOT convert education entries (such as degrees or university attendance) or project entries (such as academic, personal, or open-source projects) into work experience entries. Keep work experience mapping strictly 1-to-1 with the original parsed experience. All experience entries in the output must correspond to actual work experiences/jobs from the original resume's experience list, preserving their exact company name and date range.
